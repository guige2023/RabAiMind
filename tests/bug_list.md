## 缺陷处理流程

```
发现缺陷 → 提交缺陷 → 确认缺陷 → 分配修复 → 修复验证 → 关闭缺陷
    │           │           │           │           │
    ▼           ▼           ▼           ▼           ▼
  任何人      填写缺陷    产品确认    开发修复    测试验证    完成
  发现        报告        优先级      人员        通过
```

## 如何添加新缺陷

### 规范流程

1. **发现** - 在开发/测试中发现问题
2. **记录** - 按下方模板录入 `tests/bug_list.md`
3. **分配** - 指派给相应开发人员
4. **修复** - 开发人员修复代码
5. **验证** - 测试人员验证修复有效
6. **关闭** - 将状态改为 Closed，填写修复日期

### 新缺陷 ID 分配规则

- 现有最大 ID 为 BUG-008
- 下一个缺陷从 BUG-009 开始
- 格式：`BUG-XXX`（三位数字，不足补零）

### 状态流转

```
Open → In Progress → Closed
         ↓
       Fixed（中间状态，表示代码已修复待验证）
```

### 优先级判定

| 优先级 | 场景 |
|--------|------|
| P0 | 核心功能完全不可用、数据丢失风险、安全漏洞 |
| P1 | 重要功能受损、有可行 workaround |
| P2 | 功能部分受损、用户体验问题 |
| P3 | 轻微问题、不影响主流程 |
